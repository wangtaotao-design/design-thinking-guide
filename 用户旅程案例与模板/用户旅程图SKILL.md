# 高质量 Markdown 用户旅程生成器

## Overview

本技能帮助产品经理/设计师/需求分析师快速生成**标准化、高质量的 Markdown 用户现状旅程文档**。基于统一的模板结构和格式规范，AI 主动理解业务场景后一次性生成完整初稿，用户审阅提出修改意见后输出可直接归档或导入需求系统的最终 Markdown 文档。

### 输出文档特点

1. **标准化头部信息**：产品名称、目标用户、场景、文档版本
2. **场景说明**：明确覆盖范围，快速建立上下文
3. **分阶段 + 子阶段 + Case 分支**：结构清晰，支持复杂流程
4. **用户角色重要性标注**：⭐ 数量标识用户在该阶段的核心程度
5. **所做 / 所想 / 痛点 / 需求完整链路**：基于事实推导需求
6. **统一编号的「点子」区块**：HTML 高亮框格式，便于评审和开发追溯
7. **纯 Markdown 文本**：兼容所有文档系统，无需依赖特定编辑器

## 工作流程

### 三种梳理方式差异说明

| 方式 | 特点 | AI 行为 |
|------|------|--------|
| **选项 A：用户访谈** | 基于真实用户反馈，贴近用户真实行为 | 会自动询问您上传访谈资料，AI 将基于访谈内容提取用户行为、想法和痛点 |
| **选项 B：竞品/网上资料** | 基于公开信息，学习行业最佳实践 | AI 将结合行业通用流程和最佳实践生成旅程，可指定参考竞品 |
| **选项 C：内部业务理解推演** | 基于业务逻辑和常识，快速生成框架 | AI 将基于行业常识和业务逻辑自动生成完整旅程，无需额外资料 |

### 【步骤 0：引导信息收集】

如果用户没有提供足够信息，只输出以下引导话术，等待用户回复：

> 请告诉我以下关键信息（标注「可选」的可以不填）：
> 1. **产品类型+名称**：
> 2. **目标用户**：（如涉及多角色请列出）
> 3. **场景名称**：
> 4. **场景描述**：（可选，描述本次梳理的业务背景）
> 5. **覆盖范围**：（可选，本次旅程从哪到哪）
> 6. **用户旅程梳理方式**：（可多选）
>    - 选项 A：来源于用户访谈（选择此项将自动询问您上传访谈资料）
>    - 选项 B：来源于竞品/网上资料
>    - 选项 C：基于内部业务理解推演（AI 将基于行业常识和业务逻辑生成）
> 7. **补充信息**：（可选）

---

### 【步骤 1：生成并确认旅程框架】

根据用户提供的信息，**AI 主动理解业务场景，先生成旅程框架**，包含：

- 核心用户角色清单（标注各角色在整体旅程中的重要性）
- 场景说明（一句话概述 + 覆盖范围）
- 阶段划分（3-6 个，只列阶段名称，不展开具体内容）

输出格式示例：

```
**核心用户**：算法工程师（⭐⭐）、数据运营（⭐）、管理员（⭐⭐）

**场景说明**：电商图片内容审核模型的训练数据管理全流程，从数据准备到模型上线。

**覆盖范围**：候选版本创建 → 数据清洗与标注 → 质量门禁 → 审批发布 → 训练执行

**阶段划分**：
- 阶段 1 / 权限开通与资源配置
- 阶段 2 / 创建训练数据域
- 阶段 3 / 训练数据候选版本准备
- 阶段 4 / 候选版本验证与发布
- 阶段 5 / 模型训练执行
- 阶段 6 / 模型上线后版本管理
```

然后输出：

> 以上是旅程框架，请确认。如需调整（增删阶段、调整顺序、补充角色等），请告诉我；如没问题，回复「继续」，我将生成完整初稿。

- 若用户回复「继续」或「没问题」→ 进入【步骤 2】
- 若用户提出修改建议 → AI 重新生成框架，**再次回到【步骤 1】**，直到用户确认后继续

---

### 【步骤 2：生成完整初稿】

框架确认后，**一次性生成完整的用户旅程初稿**，包含：

- 子阶段（如有必要则拆解）
- 用户角色与重要性标注
- 所做 / 所想 / 所说
- 痛点与需求
- **点子区块**（有痛点必须生成，带编号）
- **Case 分支**（主动识别场景中的分支路径）

**AI 判断原则**：
- 子阶段拆解、用户角色重要性，由 AI 根据业务常识自行判断
- 如果信息不足或存在歧义，AI 应基于合理假设生成，并在对应位置用括号标注假设（如：`（假设：审批人为此阶段 Admin 用户）`）
- 不要中途停下来询问用户，一次性输出完整初稿
- 初稿格式严格遵守下方「输出格式规范」

输出后进入【步骤 3】

---

### 【步骤 3：审阅与修订】

只输出：

> 以上是完整初稿，请审阅。
> - 如无需修改，直接回复「没问题」，我将为您生成 Markdown 文档文件
> - 如需修改，请直接告诉我修改建议（增删阶段、调整顺序、补充 Case、修正内容等）

- 若用户回复「没问题」或「生成」或「输出」→ 在当前工作目录下创建一个 `.md` 文件，文件名为 `XXXX用户旅程图.md`（`XXXX` 取场景名称或第一个阶段名称），将完整的 Markdown 文档内容写入文件，然后告知用户文件已生成及文件路径
- 若用户提出修改建议 → AI 根据建议优化后，**再次回到【步骤 3】**，呈现修改后的版本供用户确认，直到用户回复「没问题」为止

## 强制校验规则

1. **用户角色必须标注重要性**：每个阶段涉及的用户角色，用 ⭐ 标注其在该阶段的重要性（⭐⭐ 核心用户 / ⭐ 参与用户）
2. **有痛点必有点子区块**：识别到痛点后，必须在下方生成带编号的「点子」HTML 区块
3. **信息不足时主动假设**：遇到歧义或信息缺失，基于合理假设生成并在文中标注，不要停下来问用户

### Case 分支使用原则（必须严格执行）

**Case 分支的核心目的**：当某个步骤的不同结果会导致**后续流程或产品方案产生根本性差异**时，才需要拆分为 Case。Case 分支关注的是「方案设计层面的差异」，而非「操作执行层面的成败」。

**判断标准**：生成 Case 分支前，AI 必须回答以下核心问题：

> 这个分支的不同结果，是否会导致后续的**产品架构、流程设计、交互方案**产生根本性不同？

- **是** → 生成 Case 分支（如：主站点恢复后可用 vs 不可用，分别需要不同的容灾架构设计）
- **否** → 不生成 Case 分支，在「所做」或「痛点」中自然描述即可（如：创建失败弹错误提示、网络测试不通过重试，这些属于操作反馈，不涉及方案设计差异）

**不应生成 Case 的典型场景**（在「所做」或「痛点」中描述即可）：
- 操作执行成功/失败（如：创建复制对失败、同步任务中断）
- 验证通过/不通过（如：网络连通性测试不通过、数据校验不通过）
- 简单的条件判断（如：权限有无、资源是否充足）
- 错误提示类场景（如：配置冲突提示、参数校验失败提示）

**应该生成 Case 的典型场景**：
- 不同分支导致后续流程完全不同（如：回切时原站点可用 vs 不可用，分别走回切流程 vs 重建容灾架构流程）
- 不同分支需要不同的产品功能或交互方案（如：同步复制 vs 异步复制，配置界面和监控指标完全不同）
- 不同分支涉及不同的角色参与或决策链路（如：自动切换 vs 手动切换，涉及不同的审批流程）

**自检清单**（必须全部为「否」才不生成 Case）：

1. 不同分支的**后续步骤**是否不同？
2. 不同分支是否涉及**不同的用户角色**参与？
3. 不同分支是否需要**不同的界面/功能**来支撑？
4. 不同分支是否对应**不同的业务目标或结果**？

只要有一个为「是」，就必须生成 Case 分支。

## 输出格式规范（生成时必须严格遵守）

### 1. 文档头部

每个文档必须以固定头部开头：

```markdown
# 【产品名称】 - 【场景名称】用户旅程图

> 产品：【产品名称】
> 目标用户：【角色1、角色2、角色3】
> 场景：【一句话描述场景】
> 文档版本：v1.0 / YYYY-MM-DD

---

## 场景说明

【描述本次用户旅程的业务背景、解决什么问题】

**覆盖范围：** 【阶段A】 → 【阶段B】 → 【阶段C】
```

---

### 2. 阶段格式

```markdown
---

## 阶段 N / 【阶段名称】
```

- 阶段之间用 `---` 分隔线隔开
- 标题使用二级标题 `##`
- 阶段序号从 1 开始

---

### 3. 子阶段格式

```markdown
### N.M 【子阶段名称】
```

- 标题使用三级标题 `###`
- 编号格式为 `阶段编号.子阶段序号`，如 `1.1`、`2.3`
- 同一阶段内的子阶段按顺序编号，便于引用和讨论

---

### 4. 用户角色

```markdown
**【用户】** ⭐⭐ **【角色名称】**（【补充说明，可选】）
```

- ⭐⭐ ：该阶段的核心用户（主要执行者）
- ⭐ ：该阶段的参与用户（配合或受影响）
- 如果不需要区分重要性，可统一使用 ⭐⭐

---

### 5. 所做 / 所想 / 痛点 / 需求

**单【所做】时：**

```markdown
**【所做】** 【用户具体做了什么，描述要具体到操作粒度，但不要细化到界面点击】

**【所想】** 【用户的真实想法，需要能交待因果关系，不要只写结论】
**【所说】** 【用户的真实表达，同一所做下可以有多个所说，用分号分隔】

**【痛点】** 【具体是哪个用户的痛点，以及造成痛点的原因和结果】

**【需求】** 【基于痛点提炼的需求，描述用户需要什么，而不是具体方案】
```

**多【所做】时（同一子阶段内有多个所做）：**

【所做】加序号，其附属的【所想】【所说】【痛点】【需求】【点子】紧跟在对应【所做】下方，使用空行分隔：

```markdown
**【所做 1】** 【第一个所做】

**【所想】** 【该所做下的想法】
**【所说】** 【该所做下的表达】
**【痛点】** 【该所做下的痛点】
**【需求】** 【该所做下的需求】

> 💡 **【点子】**
>
> **需求简称：** 需求的具体描述

**【所做 2】** 【第二个所做】

**【所想】** 【该所做下的想法】
```

格式要点：
- **所做**：蓝色语义，描述具体行为
- **所想**：绿色语义，描述内心想法
- **所说**：青色语义，描述用户的真实表达
- **痛点**：橙色语义，描述遇到的问题
- **需求**：紫色语义，描述需要被满足什么
- 四者层级关系：所做 →（所想、所说、痛点 → 需求）
- 单【所做】时，附属内容紧跟【所做】下方，使用空行分隔
- 多【所做】时，【所做】加序号，附属内容紧跟对应【所做】下方，使用空行分隔
- 一个「所做」下可以跟多个「所想」或多个「所说」，多条之间用分号分隔
- 一个「所做」下可以跟「痛点」，「痛点」下必须跟「需求】

---

### 6. 点子区块（核心格式）

**核心概念：** "点子"不是"功能"，团队成员一起头脑风暴能满足场景下用户的点子而非具体的功能，点子能带来相互的启发，而功能会限制住大家的。点子应描述用户能获得的价值和体验，而非具体的技术实现方式。

当识别到痛点/需求后，必须在相关段落下方生成统一格式的「点子」区块：

```markdown
> 💡 **【点子】**
>
> **需求简称：** 需求的具体描述内容
> **需求简称：** 需求的具体描述内容
```

规范要点：
- 使用 Markdown 引用块 `>` 包裹，左侧自动生成蓝色竖线标识
- 必须以 `💡 **【点子】**` 开头
- **不要编号**，直接以 `**需求简称：**` 加粗概括开头，后跟具体描述
- 点子区块紧跟在对应的「所做/所想/痛点/需求」段落之后，与下一个子阶段之间**不需要额外分隔线**
- **正确示例**：运维人员可以在一个面板上看到所有数据库的健康状态概览，无需切换多个页面
- **错误示例**：开发统一监控仪表盘功能

---

### 7. Case 分支格式

当同一子阶段存在不同结果路径时使用：

```markdown
### Case1: 【分支名称，如"审阅通过"】

**【所做】** 【该分支下的用户行为】

### Case2: 【分支名称，如"审阅不通过"】

**【用户】** ⭐ 【角色名称】
**【所做】** 【该分支下的用户行为】
**【所想】** 【该分支下的用户想法】
```

- Case 与「所做」平级，放在子阶段下
- Case 下可以继续有「所做」「所想」「痛点」「点子区块」
- 如果 Case 分支较简单，也可以直接在阶段下使用（不套子阶段）
- **编号规则**：不同 Case 分支下的子阶段各自独立编号，均从该阶段的起始序号开始（如都从 `1.1`、`1.2` 起编），不跨 Case 连续编号

---

### 8. 分隔线使用

- 阶段与阶段之间：使用 `---`
- 阶段内的不同子阶段之间：使用 `---`
- 同一子阶段内【所做】的附属内容之间：**使用空行分隔**
- 点子区块与下一个子阶段之间：**不需要额外分隔线**（引用块本身有视觉区隔效果）

---

## 节点定义规范

| 节点类型 | 说明 | 是否必填 | Markdown 格式 |
|---------|------|----------|--------------|
| 阶段 | 最粗粒度的划分，每个阶段对应一个大步骤 | ✅ 必填 | `## 阶段 N / 名称` |
| 子阶段 | 将阶段拆解为更小的步骤 | ⚪ 可选 | `### N. 名称` |
| Case 分支 | 存在不同分支情况时使用 | ⚪ 可选 | `### CaseN: 名称` |
| 用户 | 执行该阶段/子阶段的用户角色 | ✅ 必填 | `**【用户】** ⭐⭐ **名称**` |
| 所做 | 用户具体做了什么，要具体明确 | ✅ 必填 | `**【所做】** 内容`（单）/ `**【所做 N】** 内容`（多） |
| 所想/所说 | 用户的真实想法和表达，需要完整交待因果关系 | ⚪ 可选 | `**【所想】** 内容` |
| 痛点 | 用户遇到的具体痛点及原因 | ⚪ 可选（识别到则需要） | `**【痛点】** 内容` |
| 需求 | 基于痛点提炼的需求和改进点子 | ⚪ 可选（有痛点则需要） | `**【需求】** 内容` |
| 点子 | 面向开发/测试可执行的需求条目 | ⚪ 可选（有需求则需要） | `> 💡 **【点子】**` 开头，下方用 `>` 包裹每条需求 |

### 撰写要求

- **所做**：不需要细致到界面操作粒度，但要比功能模块名更具体。例如可以写成 `"定义告警规则"`，不能写成 `"点击设置按钮"` `"勾选复选框"`
- **所想/所说**：必须能交待因果关系。同一【所做】下多个【所想】或【所说】用分号分隔。❌ 错误示例：`"希望系统好用"`；✅ 正确示例：
  ```
  **【所想】** 告警通知太多反而掩盖了核心问题，希望能区分紧急程度；发现问题时已经对用户造成影响，平时很难做到预警
  ```
- **痛点**：必须说明是哪个用户的痛点，以及造成痛点的原因和结果/表现
- **需求**：使用标准句式：`【特定用户角色】需要一种方式来【做什么事情】，这样可以【直接受益】`
- **点子区块**：面向开发/测试可执行的需求条目，可包含功能点、规则说明、验收标准、改进点子

## 示例片段

以下是一个符合规范的片段示例，生成时必须达到类似质量：

```markdown
## 阶段 2 / 训练数据候选版本准备

### 2.1 算法工程师发起候选版本

**【用户】** ⭐⭐ **算法工程师**

**【所做 1】** 收到业务需求——暴力图识别率不达标，决定启动新一轮训练数据准备，从当前稳定训练集主线拉出一个候选版本

**【所做 2】** 编写标注规范，明确各类别的判定标准和边界案例处理方式，交给数据运营一并传达给标注团队

**【所想】** 在候选版本里改，不会动现在线上跑的那份训练集，这轮数据准备出了什么问题也不会影响生产

**【痛点】** 补数、重算直接改生产训练集，下游训练任务读到半成品数据。新一轮数据准备没有隔离环境，写入会直接影响正在使用的生产训练集。

**【需求】** 需要候选版本机制，让新训练集和当前生产训练集隔离，所有变更先在候选区发生

> 💡 **【点子】**
>
> **创建分支：** 从当前主线一键拉出候选分支
```
